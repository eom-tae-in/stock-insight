다음 PRD를 기준으로 TRD(Technical Requirements Document)를 생성해주세요.

검토 및 설계 대상 PRD는 본 요청에 함께 제공된 문서를 기준으로 합니다.

---

[입력 문서]
- 제공된 PRD 문서 1개

---

[추가 기술 조건]
- 아래 조건이 명시된 경우 반드시 반영하세요.
- 명시되지 않은 항목은 최소한의 가정만 사용하고, 그 가정은 TRD에 "(가정)"이라고 표시하세요.

---

[작성 지시]

- 반드시 제공된 PRD를 기준으로만 작성하세요.
- PRD에 없는 기능은 절대 추가하지 마세요.
- PRD의 기능 ID(Fxxx), 페이지, 사용자 흐름, 핵심 데이터 개체를 기준으로 TRD를 구성하세요.

- 모든 기능 ID(Fxxx)는 반드시 하나의 도메인에 속해야 합니다.
- 모든 기능 ID는 API / 서비스 로직 / 데이터 처리 / 페이지 흐름 중 하나 이상과 연결되어야 합니다.

- 모든 페이지는 반드시 하나 이상의 도메인과 연결되도록 작성하세요.
- 페이지 설명은 반드시 API / 데이터 흐름 / 처리 로직으로 변환하세요.

- API 목록만 나열하지 말고, 반드시 서비스/비즈니스 로직과 주요 처리 시퀀스까지 포함하세요.

---

[핵심 설계 규칙]

1. 도메인 중심 설계
- 기능을 도메인 단위로 묶으세요.
- 기준:
  - 같은 데이터 개체
  - 같은 책임
  - 같은 사용자 흐름
- 도메인은 2~5개 수준 유지

2. API 최소화 원칙
- “이 API 없으면 기능 구현 불가능한가?” 기준으로만 생성
- 불필요한 CRUD 분리 금지
- 조회/검증용 API 남발 금지

3. PRD 애매성 처리
- PRD 범위까지만 해석
- 반드시 필요한 경우만 "(가정)" 사용
- 기능 확장 금지
- 애매하면 줄이기

4. MVP 기준
- 과설계 금지
- MSA / CQRS / 이벤트 구조 금지

---

[정합성 요구사항]

- PRD의 모든 기능은 TRD에 연결되어야 합니다.
- 페이지 ↔ API ↔ 데이터 흐름이 일관되어야 합니다.
- 신규 기능 ID 생성 금지
- PRD와 모순되는 흐름 금지
