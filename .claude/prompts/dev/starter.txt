[입력 문서]
- TRD 문서 1개

---

[현재 프로젝트 상태]
- 아래 중 하나로 가정하고 작업하세요:

1) 신규 프로젝트 (아직 코드 없음)
2) 기존 스타터 존재 (보일러플레이트/데모 코드 포함)

※ 명확하지 않을 경우, 두 경우를 모두 고려하여 판단 근거를 제시하세요.

---

[추가 조건]
- MVP 기준으로만 구성 (과설계 금지)
- TRD에 없는 기능/구조는 추가하지 마세요
- 기술은 반드시 TRD 요구사항을 근거로 선택하세요
- 특정 프레임워크를 기본값으로 가정하지 마세요
- 최신 정보는 참고하되, 안정성과 공식 권장 방식을 우선하세요

[작업 지시]

다음 순서를 반드시 따르세요:

---

1️⃣ TRD 해석

- 제품 유형
- 핵심 기능
- 주요 사용자 흐름
- 기능 ID 구조
- 필요한 기술 요소 (Frontend / Backend / DB / Auth 등)
- 비기능 요구사항
- MVP 범위
- 불명확한 부분은 "(가정)"으로 명시

---

2️⃣ 기술 선택

각 영역별 기술을 선택하고 이유를 명확히 작성하세요:

- Frontend:
- Backend:
- Database:
- Auth:
- 기타:

각 기술에 대해:
- 선택 이유
- TRD와의 연결 근거
- 대안 기술이 아닌 이유

---

3️⃣ 최신 정보 보강

선택한 기술 기준으로 다음을 정리하세요:

- 공식 초기화 방법
- 권장 안정 버전
- 필수 설정 방식
- deprecated 여부
- 주의사항

※ 최신 트렌드가 아니라 “안전한 시작” 기준으로 판단하세요

---

4️⃣ 초기화 전략 수립

아래 중 어떤 방식인지 판단하고 이유를 작성하세요:

- 신규 프로젝트 생성
- 기존 스타터 정리

그리고 다음을 정의하세요:

- 사용할 initializer (있는 경우)
- 제거 대상
- 유지 대상
- 구조 정리 방향

---

5️⃣ 프로젝트 구조 정의

초기화 이후 남길 최소 구조를 정의하세요.

요구사항:
- 단순하고 확장 가능해야 함
- TRD 기능을 담을 수 있어야 함
- roadmap으로 바로 이어질 수 있어야 함

디렉토리 구조와 각 역할을 설명하세요.

---

6️⃣ 정리 및 초기화 결과

다음을 명확히 구분해서 작성하세요:

✅ 생성/유지
❌ 제거
⚠️ 보류

---

7️⃣ 검증

다음 항목을 기준으로 검증하세요:

- 실행 가능 여부
- 빌드 가능 여부
- 타입 오류 여부
- 의존성 문제 여부
- 구조와 TRD 정합성

---

8️⃣ roadmap 인계 기준

다음 단계에서 roadmap을 만들 수 있도록:

- 어떤 단위로 task를 쪼개야 하는지
- 선행 작업 / 후행 작업 기준
- 구현 시작 지점
