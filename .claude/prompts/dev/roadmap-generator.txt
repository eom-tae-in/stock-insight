[입력 문서]
- PRD 문서 1개
- TRD 문서 1개

[추가 참고 자료]
- starter 결과 문서가 있다면 참고할 수 있습니다.
- 다만 starter 결과 문서 유무와 관계없이, 반드시 현재 프로젝트 구조, 핵심 설정 파일, package.json 등을 직접 분석하여 현재 시작 상태를 파악하세요.
- roadmap은 문서만 기준으로 작성하지 말고, starter 이후 실제 코드베이스 상태를 우선 반영하세요.

---

[로드맵 생성 목적]
이 로드맵은 설계 문서를 다시 요약하는 문서가 아닙니다.
PRD/TRD와 starter 이후의 현재 프로젝트 상태를 기준으로,
실제 구현 가능한 순서와 작업 단위로 분해한 실행 계획이어야 합니다.

특히 다음을 반드시 지키세요:
- TRD만 기준으로 추상적인 로드맵을 작성하지 마세요.
- 실제 디렉토리 구조와 현재 코드 상태를 기반으로 task를 설계하세요.
- 존재하지 않는 구조를 임의로 가정하지 마세요.
- starter 단계에서 이미 정리된 구조와 기술 선택을 다시 설계하지 마세요.

- TRD에 없는 기능을 추가하지 마세요

- starter 결과 문서가 있다면 참고하되, 실제 현재 프로젝트 구조를 우선 기준으로 삼으세요
- 작업 단위는 실제 구현 가능한 수준으로 쪼개세요

- roadmap은 바로 구현에 들어갈 수 있어야 합니다
- 각 task는 현재 구조의 어느 레이어/폴더/파일 영역에서 시작해야 하는지 드러나야 합니다

[작업 지시]

다음 순서를 반드시 따르세요.

---

1️⃣ 입력 해석

다음을 먼저 정리하세요:

- PRD 기준 핵심 사용자 흐름
- TRD 기준 기능 ID 및 기술 구조
- starter 이후 현재 프로젝트 시작 상태
- 현재 프로젝트 루트 구조 및 주요 디렉토리 역할
- 현재 이미 준비된 것 / 아직 구현되지 않은 것
- MVP 범위
- 불명확한 부분은 "(가정)"으로 명시

주의:
- 반드시 현재 프로젝트 구조를 먼저 분석하세요.
- PRD/TRD만 보고 roadmap을 작성하지 마세요.
- starter가 만든 실제 시작 상태와 문서 내용이 다르면 그 차이를 명시하세요.

---

2️⃣ 구현 전략 요약

다음을 정리하세요:

- 전체 구현 방향
- 어떤 순서로 구현해야 하는지에 대한 큰 흐름
- 선행 작업이 필요한 영역
- 병렬 가능한 영역
- MVP 이후로 미뤄도 되는 영역

이때 반드시:
- 현재 구조 기준으로 어디서부터 구현이 시작될지 설명하세요
- 선행 기반 작업과 기능 구현 작업을 구분하세요
- 이미 starter에서 준비된 영역은 다시 task로 과도하게 쪼개지 마세요

---

3️⃣ Phase 설계

roadmap을 Phase 단위로 나누세요.

각 Phase에는 반드시 다음이 포함되어야 합니다:

- Phase 이름
- 목표
- 선행 조건
- 포함 task
- 완료 기준
- 다음 Phase로 넘길 기준

Phase는 프로젝트 성격에 맞게 나누되,
단순 나열이 아니라 실제 구현 순서가 드러나야 합니다.

추가 기준:
- starter에서 끝난 작업은 roadmap phase에 다시 넣지 마세요
- phase는 현재 구조에서 자연스럽게 이어지는 구현 흐름이어야 합니다
- 공통 기반 → 핵심 도메인/데이터 → 핵심 사용자 흐름 → 부가 기능 → 검증/정리 순으로 생각하되, 프로젝트 특성에 맞게 조정하세요

---

4️⃣ Task 분해

각 Phase 안의 task를 실제 구현 가능한 단위로 작성하세요.

각 task에는 다음을 포함하세요:

- Task ID
- Task 이름
- 목적
- 작업 내용
- 관련 기능 ID
- 관련 구조(페이지, API, 서비스, DB, 컴포넌트, 상태관리, 설정 등)
- 선행 작업
- 완료 기준

주의:
- "로그인 구현", "대시보드 구현"처럼 뭉뚱그리지 마세요
- 한 번에 구현 가능한 수준으로 나누세요
- 코드 생성 에이전트가 바로 받을 수 있는 수준으로 구체화하세요
- 각 task는 실제로 손대야 하는 구조가 드러나야 합니다
- 존재하지 않는 디렉토리나 레이어를 임의로 전제하지 마세요

---

5️⃣ 의존성 및 우선순위 정리

다음을 구분하세요:

- 반드시 먼저 해야 하는 작업
- 병렬 가능한 작업
- 후순위 작업
- MVP 포함 / MVP 제외

이때 다음 기준을 반영하세요:
- 데이터 구조/인증/공통 기반처럼 후속 구현의 기반이 되는 작업은 선행으로 배치하세요
- 현재 구조상 이미 준비된 기반은 재작업하지 마세요
- 리스크가 큰 작업은 너무 뒤로 미루지 마세요

---

6️⃣ 검증 기준 정리

각 Phase 또는 핵심 Task에 대해 다음 중 필요한 검증 기준을 포함하세요:

- 페이지 진입 가능
- API 요청/응답 정상
- DB 저장/조회 정상
- 인증 흐름 정상
- 핵심 시나리오 수동 검증 완료
- 최소 테스트 통과

추가로:
- 검증 기준은 추상적으로 쓰지 말고, 실제 완료 여부를 판단할 수 있게 작성하세요
- 가능한 경우 현재 구조 기준으로 어떤 레벨에서 검증하는지 드러내세요

---

7️⃣ 구현 시작 가이드 작성

마지막에 반드시 정리하세요:

- 어디부터 구현을 시작해야 하는지
- 첫 번째 task는 무엇인지
- 그 다음 어떤 순서로 이어지는지
- 구현 시 가장 먼저 부딪힐 리스크는 무엇인지

추가로:
- 첫 시작 지점은 현재 구조 기준으로 가장 자연스러운 위치여야 합니다
- starter 이후 비어 있는 핵심 영역부터 시작할지, 데이터/도메인 선행 작업부터 시작할지 명확히 판단하세요
