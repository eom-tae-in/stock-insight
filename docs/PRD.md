# StockInsight PRD

## 1. 제품 정의

StockInsight는 개인 투자자가 특정 종목의 가격 흐름과 대중 검색 관심도 변화를 함께 분석하는 웹 애플리케이션이다. 사용자는 종목과 키워드를 저장하고, 기간/지역/검색 타입별 트렌드 분석과 종목 오버레이를 비교하며, 차트와 데이터를 다운로드할 수 있다.

현재 프로젝트는 기존 종목 분석 MVP에서 키워드 중심 분석 도구로 확장되는 중간 단계에 있다. 이 PRD는 앞으로의 기준 제품 범위를 정의하고, 이후 TRD와 ROADMAP의 기준 문서로 사용한다.

## 2. 목표

- 저장된 종목과 키워드 분석을 사용자별로 안전하게 관리한다.
- Yahoo Finance 주가 데이터와 Google Trends 관심도 데이터를 동일한 주간 축으로 정렬한다.
- 키워드 분석은 `키워드 + 지역 + 기간 + 검색 타입` 조합별로 독립 저장한다.
- 각 키워드 분석에 여러 종목을 오버레이하여 상관 흐름을 비교한다.
- 다운로드, 재조회, 삭제, 정렬 등 반복 분석에 필요한 작업을 안정적으로 제공한다.
- 코드와 DB 스키마의 단일 모델을 확정해 유지보수 가능한 구조로 전환한다.

## 3. 비목표

- 실시간 시세 스트리밍
- 매매 추천 또는 투자 자문
- 뉴스/소셜 감성 분석
- 포트폴리오 수익률 관리
- 다중 사용자 협업, 공유 링크, 댓글
- 모바일 앱 네이티브 배포

## 4. 사용자

주 사용자는 개인 투자자 또는 리서치 목적으로 종목과 검색 관심도 관계를 반복 확인하는 단일 사용자다. 제품은 Supabase Auth 기반 다중 계정 구조를 가진다. 모든 저장 데이터는 사용자별로 격리되어야 한다.

## 5. 핵심 사용자 여정

### 5.1 인증

1. 사용자는 회원가입 또는 로그인한다.
2. 인증되지 않은 사용자는 보호된 앱 경로에 접근할 수 없다.
3. 로그인 후 대시보드로 이동한다.

### 5.2 종목 분석

1. 사용자는 종목 검색 페이지에서 ticker 또는 회사명을 검색한다.
2. 앱은 ticker를 확정하고 Yahoo Finance에서 5년 주간 주가를 수집한다.
3. 종목 메타데이터와 주가 시계열을 저장한다.
4. 사용자는 종목 상세 화면에서 지표, 차트, 원본 테이블을 확인한다.
5. 필요 시 종목 데이터를 삭제하거나 최신 데이터로 갱신한다.

### 5.3 키워드 분석

1. 사용자는 키워드 검색 화면에서 키워드, 지역, 기간, 검색 타입을 선택한다.
2. 앱은 Google Trends 데이터를 수집하고 MA13, YoY를 계산한다.
3. 동일 키워드라도 지역/기간/검색 타입이 다르면 별도 분석으로 저장한다.
4. 사용자는 분석 상세 화면에서 트렌드 차트와 지표를 확인한다.
5. 사용자는 분석에 종목 오버레이를 추가하고 순서를 조정한다.
6. 필터를 바꾸면 해당 조건 조합의 분석과 오버레이만 표시된다.

### 5.4 내보내기

1. 사용자는 종목 또는 키워드 분석 화면에서 Excel을 다운로드한다.
2. 사용자는 차트 PNG를 다운로드한다.
3. 내보내기 데이터는 화면에 표시된 조건과 동일해야 한다.

## 6. 기능 요구사항

### F001. 인증 및 사용자 격리

- Supabase Auth로 이메일 기반 회원가입, 로그인, 로그아웃을 지원한다.
- 모든 저장 데이터는 `auth.uid()` 기준으로 격리한다.
- API Route는 공통 인증 헬퍼를 통해 사용자 인증을 검증한다.
- RLS와 코드 레벨 필터를 함께 사용해 소유권 경계를 명확히 한다.

### F002. 종목 검색 및 저장

- ticker 또는 회사명 기반 검색을 지원한다.
- 사용자가 선택한 ticker에 대해 Yahoo Finance 데이터를 수집한다.
- 동일 사용자의 동일 ticker는 중복 생성하지 않고 갱신한다.
- 종목 저장 단위는 사용자별 ticker다.

### F003. 종목 시계열 수집

- 최근 5년 주간 OHLCV 데이터를 수집한다.
- 날짜는 ISO week 시작일 기준 `YYYY-MM-DD`로 정규화한다.
- 저장 시 `searches`에는 메타데이터를, `stock_price_data`에는 시계열을 저장한다.
- 같은 `search_id + date`는 upsert한다.

### F004. 종목 상세 분석

- 현재가, 이전 종가, MA13, YoY, 52주 최고/최저를 표시한다.
- 주가 차트, MA13 차트, 테이블 뷰를 제공한다.
- 저장된 데이터가 없거나 접근 권한이 없으면 명확한 404/권한 오류를 표시한다.

### F005. 키워드 검색 및 분석 생성

- 입력값: keyword, region, period, search_type.
- region 후보: `GLOBAL`, `US`, `KR`, `JP`, `CN`.
- period 후보: `1Y`, `3Y`, `5Y`.
- search_type 후보: `WEB`, `YOUTUBE`.
- 같은 사용자, 같은 keyword, region, period, search_type 조합은 한 개의 분석으로 유지한다.

### F006. Google Trends 수집

- Google Trends 수집 경로는 하나의 서비스 레이어로 통합한다.
- 로컬과 배포 환경에서 동일한 응답 계약을 유지한다.
- API 실패, 차단, 데이터 없음, 타임아웃을 구분해 사용자에게 재시도 가능한 오류로 표시한다.
- 수집 결과는 `date`, `value`, `ma13Value`, `yoyValue` 형태로 저장한다.

### F007. 키워드 분석 상세

- 분석 조건과 트렌드 시계열을 표시한다.
- 최신 트렌드 값, MA13, YoY를 요약한다.
- 기간/지역/검색 타입 변경 시 해당 조합의 분석으로 이동하거나 새로 생성한다.
- 조건 변경으로 다른 분석을 볼 때 기존 분석의 오버레이가 섞이면 안 된다.

### F008. 종목 오버레이

- 키워드 분석 하나에 여러 종목을 오버레이할 수 있다.
- 오버레이는 `analysis_id` 기준으로 저장한다.
- 같은 분석 안에서 같은 종목은 중복 추가할 수 없다.
- 사용자는 오버레이 순서를 변경하고 삭제할 수 있다.
- 오버레이 시계열은 트렌드 차트 날짜 축과 정렬 가능한 형태여야 한다.

### F009. 대시보드

- 저장된 종목과 저장된 키워드 분석으로 이동할 수 있는 진입점을 제공한다.
- 최근 조회/수정 순으로 정렬한다.
- 빈 상태, 로딩 상태, 오류 상태를 제공한다.

### F010. 다운로드

- 종목 분석 Excel: 종목 메타데이터, 주가 시계열, 계산 지표.
- 키워드 분석 Excel: 조건, 트렌드 시계열, 오버레이 시계열.
- PNG: 화면에 표시된 차트를 이미지로 저장한다.

## 7. 데이터 요구사항

### 7.1 단일 진실 원천

새로운 기준 모델은 다음과 같다.

- 종목: `searches`, `stock_price_data`
- 키워드 컨테이너: `keywords`
- 조건별 키워드 분석: `keyword_analysis`
- 분석별 종목 오버레이: `keyword_stock_overlays`
- 오버레이 차트 시계열: `overlay_chart_timeseries`

`keyword_searches`, `keyword_chart_timeseries`, `keyword_temporary_overlays`는 마이그레이션 호환 레이어로만 취급하고, 최종 아키텍처에서는 제거 또는 읽기 전용 마이그레이션 대상으로 정리한다.

### 7.2 핵심 관계

```
auth.users
  └─ searches
       └─ stock_price_data

auth.users
  └─ keywords
       └─ keyword_analysis
            ├─ keyword_stock_overlays
            │    └─ overlay_chart_timeseries
            └─ trends_data JSONB
```

### 7.3 소유권

- `searches.user_id = auth.uid()`
- `keywords.user_id = auth.uid()`
- `keyword_analysis`는 `keywords.user_id`를 통해 소유권을 검증한다.
- `keyword_stock_overlays`는 `keyword_analysis -> keywords.user_id`를 통해 소유권을 검증한다.

## 8. 품질 요구사항

- `npm run typecheck`, `npm run lint`, `npm run build`가 통과해야 한다.
- 핵심 계산 함수는 단위 테스트를 가진다.
- API Route는 인증, 입력 검증, 성공, 실패 케이스 테스트를 가진다.
- 주요 사용자 여정은 Playwright E2E 테스트로 검증한다.
- DB 스키마와 TypeScript 타입은 같은 모델을 표현해야 한다.
- 서버 로그에 민감 정보가 남으면 안 된다.
- 프로덕션 UI는 내부 에러 메시지를 그대로 노출하지 않는다.

## 9. 성공 기준

- 키워드 분석 데이터 모델이 하나로 통일된다.
- Trends 수집 경로가 하나로 정리된다.
- 사용자별 데이터 격리가 코드와 RLS 양쪽에서 검증된다.
- 종목 분석, 키워드 분석, 오버레이 추가/삭제/정렬, 다운로드 플로우가 안정적으로 동작한다.
- 신규 기능 개발자가 PRD/TRD/ROADMAP만 보고 작업 범위와 구조를 이해할 수 있다.
